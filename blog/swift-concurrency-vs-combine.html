Why I Chose Swift Concurrency Over Combine

February 28, 2026  ·  6 min read

Combine felt like the obvious choice when I started building async networking for Spectral. Twelve months later I rewrote the whole layer using async/await and never looked back. Here's the honest comparison.

Why I Started With Combine

When I first built Spectral's image-fetching pipeline, Combine was the idiomatic Apple answer to reactive networking. It plugged directly into SwiftUI via ObservableObject, it had first-party support, and there was plenty of documentation. The pipeline looked clean on paper: fetch a URL, decode the response, publish to a @Published property, done.

For a while it worked. But as Spectral grew — offline caching, background refresh, error recovery, pagination — the Combine chains grew with it. And that's where the trouble started.

The Problem With Combine at Scale

Combine is powerful, but it encourages you to express control flow as a chain of operators rather than as readable procedural code. Here's a simplified version of what Spectral's wallpaper-fetch pipeline looked like after six months:

URLSession.shared.dataTaskPublisher(for: url)
    .tryMap { data, response -> Data in
        guard let http = response as? HTTPURLResponse,
              http.statusCode == 200 else {
            throw FetchError.badStatus
        }
        return data
    }
    .decode(type: WallpaperPage.self, decoder: JSONDecoder())
    .receive(on: DispatchQueue.main)
    .catch { _ in Just(WallpaperPage.empty) }
    .flatMap { page in self.cacheImages(for: page) }
    .sink { ... }
    .store(in: &cancellables)

This is readable. The problem is what it becomes once you add retry logic, conditional caching, and branching error paths. The chain grows horizontally and the logic fragments across closures. When a bug appears, tracing it through the operator chain is genuinely painful.

The Rewrite With async/await

Swift Concurrency lets you write the same logic as sequential, readable code:

func fetchWallpapers(from url: URL) async throws -> WallpaperPage {
    let (data, response) = try await URLSession.shared.data(from: url)
    guard let http = response as? HTTPURLResponse, http.statusCode == 200 else {
        throw FetchError.badStatus
    }
    let page = try JSONDecoder().decode(WallpaperPage.self, from: data)
    await cacheImages(for: page)
    return page
}

The branching, the error propagation, the ordering — all of it is immediately obvious. A junior dev reading this for the first time understands it in 30 seconds. The Combine version requires knowing the operator vocabulary.

Structured Concurrency for Parallel Fetches

Spectral fetches multiple image categories in parallel on launch. With Combine this meant a Publishers.Zip or Publishers.MergeMany — both of which have quirks around error handling and cancellation. With structured concurrency:

async let featured = fetchWallpapers(from: .featured)
async let trending = fetchWallpapers(from: .trending)
async let recent = fetchWallpapers(from: .recent)

let (f, t, r) = try await (featured, trending, recent)

If any one of these throws, the others are cancelled automatically. No cancellables bag. No .store(in:). The structured concurrency model handles it.

What Combine Is Still Good For

I'm not arguing Combine is bad — it's a well-designed framework. It still shines for binding UI state to streams of values, especially with SwiftUI's onReceive. I kept a small Combine layer in Spectral for things like debouncing search input and observing NotificationCenter events. For those use cases, a short operator chain is cleaner than an AsyncSequence.

Use async/await for imperative, stateful workflows. Use Combine for declarative, value-stream bindings. The two coexist fine — you don't have to pick one.

The Bottom Line

If you're starting a new networking layer today, start with Swift Concurrency. You'll write less code, catch errors more naturally, and your teammates will thank you during code review. Migrate to Combine operators only where you genuinely need reactive pipelines — not because it feels more "Apple-idiomatic."

The rewrite took me a week. The clarity dividend has paid back every hour of it.
